System Design and Integration

A complex robotics application like our capstone project requires a well-designed software architecture. In this section, we will lay out the system design for our autonomous household assistant, detailing the various components and how they are integrated.

We will be using a modular, ROS 2-based architecture, where each major function is handled by a dedicated node.

Component Descriptions​

Let's break down the role of each component in the architecture.

1. User Interface​

  • Voice Input: A simple node that captures audio from a microphone and converts it to text using a speech-to-text engine.
  • NLU Pipeline: This node takes the text command and uses a pre-trained language model to extract the user's intent and the key entities. For our project, the output would be something like:
    • Intent: pick_and_place
    • Object: water_bottle
    • Source: table
    • Destination: fridge

2. VLA Model​

  • Task Planner: This is the high-level "brain" of the robot. It receives the structured command from the NLU pipeline and the current state of the world from the perception node. It then breaks down the task into a sequence of sub-tasks, such as navigate_to(table), pick(water_bottle), navigate_to(fridge), place(in_fridge).
  • Action Generation: This component takes a sub-task from the task planner and generates a sequence of low-level actions to achieve it. For example, for the pick(water_bottle) sub-task, it would generate a series of arm movements to reach for, grasp, and lift the bottle.

3. ROS 2 System​

  • Perception Node: This node is responsible for processing the sensor data from the simulator. It will use the RGB-D camera data to run an object detection model to find the water bottle, table, and fridge. It will publish the detected objects and their poses to the rest of the system.
  • Navigation Node (Nav2): We will use the standard ROS 2 Nav2 stack for navigation. This node will take a goal location from the task planner and generate a path to it, avoiding obstacles along the way.
  • Manipulation Node (MoveIt): For manipulation, we will use the MoveIt 2 framework. This node will take a target pose for the robot's arm from the action generation component and plan a collision-free trajectory to reach it.
  • Robot Controller: This is the low-level controller that interfaces with the simulated robot. It will use ros2_control to send commands to the robot's joints and publish the robot's state (joint angles, etc.).

4. Simulation​

  • Gazebo/Isaac Sim: The simulator provides the virtual environment, the robot model, and the sensor data. It communicates with the ROS 2 system through the ROS bridge.

Integration and Communication​

The components will communicate with each other using a combination of ROS 2 topics, services, and actions.

  • The NLU pipeline will send the structured command to the task planner via a topic.
  • The perception node will publish the detected objects to a topic.
  • The task planner will call the navigation and manipulation nodes using actions, as these are long-running tasks.
  • The action generation component will send low-level commands to the robot controller via topics.

This modular design allows us to develop and test each component independently before integrating them into the final system. In the next section, we will provide a step-by-step guide to implementing each of these components.
